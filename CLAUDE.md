# Webアーキテクチャ学習リポジトリ：ClaudeCode運用指針

---

## 1. このリポジトリの目的

本リポジトリは、Webフロント領域を含むWebアーキテクチャを「実装者」ではなく「設計レビューを行うTechLead」の視点で理解するための学習プロジェクトです。

目標は以下を説明・判断できる状態になることです。

- ブラウザ ↔ フロント ↔ API ↔ DB の構造を説明できる
- HTTPの流れとヘッダの役割を説明できる
- Cookie / JWT の責務とリスクを語れる
- CORSがなぜ発生するかを構造的に説明できる
- CSRとSSRの設計判断を説明できる
- BFFの必要性と境界設計を語れる
- Webパフォーマンスのボトルネックを推測できる
- フロントエンド固有のセキュリティリスクを指摘できる
- 非機能観点を含めたレビューができる

機能完成は目的ではありません。
理解の深化と判断軸の獲得が目的です。

---

## 2. 学習伴走スタイル（最重要）

このリポジトリは学習用です。

ClaudeCodeは以下を厳守してください。

- どんどん実装を進めない
- 各フェーズ開始前に「何をやるか」「なぜやるか」「何を学ぶか」を説明する
- 代替案とトレードオフを必ず提示する
- レビュー観点でのチェックポイントを提示する
- 実装後に学びを整理する
- 問題が起きた場合は切り分け思考を言語化する

常に「Reviewer視点」で伴走すること。

---

## 3. 公開リポジトリポリシー（厳守）

本リポジトリは public で公開します。

必ず以下を守ること。

- 実在する企業情報・業務情報を含めない
- APIキー・トークン・パスワードをコードに直書きしない
- 環境変数は .env 管理し、.gitignore に含める
- ダミーデータのみ使用する
- 実在サービスの内部仕様を再現しない
- 公開して問題のない設計例のみ扱う

---

## 4. 技術スタック

- Frontend: Next.js（App Router）+ TypeScript
- Backend: Spring Boot（Java）+ REST API
- DB: PostgreSQL（Docker）
- Local環境: Docker Compose（front / api / db）
- 認証は Cookieセッション → JWT の順で比較

今回の主戦場は「Web構造理解」である。

---

## 5. ローカル実行を前提とする

- 常にローカルで再現可能な構成を維持する
- Docker Composeで一発起動可能にする
- 挙動を観察しながら進める
- 意図的に壊して理解する（特にCORS・Cookie・SSR）

# 9. 段階的学習テーマ（自然な理解順）

## Phase 1: Webの最小単位を作る（認証なし）

目的：
Webアプリの全体構造を理解する。

実装内容：
- 認証なしCRUD（例：posts）
- フロントからAPI呼び出し
- DB連携
- CORSを意図的に発生させる

学ぶ観点：
- ブラウザ ↔ フロント ↔ API ↔ DB の構造
- HTTPの基本
- CORSの発生条件
- fetchの実態

---

## Phase 2: 認証導入（Cookieセッション）

目的：
Webらしさの核心である「セッション」を理解する。

実装内容：
- Spring Security導入
- ログイン / ログアウト
- Cookieセッション管理
- 認可（自分のデータのみ操作可能）

学ぶ観点：
- Cookieの挙動
- HttpOnly / Secure / SameSite
- 認証と認可の違い
- CSRFが発生する構造

---

## Phase 3: 認証方式比較（JWT）

目的：
設計レビューで必ず問われる論点を理解する。

実装内容：
- JWT導入
- 保存場所の比較
- SameSiteとの関係整理
- XSSリスク整理

学ぶ観点：
- トークン露出リスク
- Cookie vs JWT のトレードオフ
- 設計判断基準

---

## Phase 4: CSRとSSRの比較

目的：
レンダリング戦略の判断軸を持つ。

実装内容：
- CSR実装（main）
- SSR実装（feature/ssr）
- 同一機能で比較
- 初回表示速度観察

学ぶ観点：
- データ取得のタイミング
- hydrationの役割
- SEOとの関係
- 認証との整合性

---

## Phase 5: BFFアーキテクチャ

目的：
境界設計を語れるようにする。

実装内容：
- API直接アクセス構成
- Next.js側にBFFを置く構成
- CORS差分比較

学ぶ観点：
- セキュリティ境界
- トークン露出リスク
- フロントとAPIの責務分離

---

## Phase 6: パフォーマンスとキャッシュ

目的：
ボトルネックを推測できるようにする。

実装内容：
- Cache-Control調整
- ETag導入
- 負荷試験
- SSR/CSR性能比較

学ぶ観点：
- HTTPキャッシュの役割
- CDNの意味
- 計測の視点

---

## Phase 7: フロントの物理法則（最終深化）

目的：
レンダリングや非同期挙動を説明できる状態へ。

実装内容：
- 再レンダリング検証
- イベントループ実験
- 不要描画の観察
- hydrationの挙動確認

学ぶ観点：
- microtask / macrotask
- 再描画の原因
- パフォーマンス劣化の根本要因
