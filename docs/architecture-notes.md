設計判断の根拠、代替案、トレードオフを記録するドキュメントです。

## 記録方針

各設計判断について以下を記録する：

1. **決定内容** - 何を選んだか
2. **理由** - なぜそれを選んだか

## 認証方式の比較

### Cookieセッション vs JWT

| 観点 | Cookieセッション | JWT |
|------|------------------|-----|
| 状態管理 | サーバー側（Stateful） | クライアント側（Stateless） |
| スケーラビリティ | セッションストア共有が必要 | サーバー間共有不要 |
| セキュリティ | HttpOnlyで保護可能 | 保存場所次第でXSSリスク |
| 失効 | サーバーで即時失効可能 | 有効期限まで失効困難 |
| 実装複雑度 | 低い | リフレッシュ戦略が必要 |

### 設計判断基準

- **セッション推奨ケース**
  - モノリシックなアプリケーション
  - セキュリティ要件が高い（即時失効必須）
  - サーバー側で状態管理が許容される

- **JWT推奨ケース**
  - マイクロサービス間認証
  - サーバーレス環境
  - 複数ドメインでの認証共有

### Reviewer観点チェックリスト

- [ ] トークン/セッションの有効期限は適切か
- [ ] 失効の仕組みは実装されているか
- [ ] 保存場所のセキュリティは確保されているか
- [ ] CSRF対策は実装されているか（Cookieの場合）
- [ ] XSS対策は実装されているか（JWTの場合）

## レンダリング戦略の比較

### CSR vs SSR vs SSG

| 観点 | CSR | SSR | SSG |
|------|-----|-----|-----|
| 初回表示 | 遅い（JS実行後） | 速い（HTML即表示） | 最速（事前生成） |
| SEO | 不利（クローラー依存） | 有利 | 有利 |
| サーバー負荷 | 低い | 高い（リクエスト毎） | 低い（ビルド時のみ） |
| データ鮮度 | 常に最新 | 常に最新 | ビルド時点 |
| 認証との相性 | 良い | 注意が必要 | 難しい |

### 設計判断基準

- **CSR推奨ケース**
  - ダッシュボード系（認証後画面）
  - リアルタイム更新が必要
  - SEO不要

- **SSR推奨ケース**
  - ランディングページ
  - SEO重要なコンテンツページ
  - 初回表示速度が重要

- **SSG推奨ケース**
  - ブログ、ドキュメント
  - 更新頻度が低いコンテンツ
  - 大量アクセスが想定される

### Reviewer観点チェックリスト

- [ ] レンダリング戦略の選択理由は明確か
- [ ] 認証が必要なページでSSRを使う場合、認証情報の扱いは適切か
- [ ] Hydration Mismatchの可能性はないか
- [ ] キャッシュ戦略との整合性はあるか

## CORS設計

### 設定パターン

| パターン | 説明 | リスク |
|----------|------|--------|
| `*` (ワイルドカード) | 全オリジン許可 | 高（認証と併用不可） |
| 特定オリジン指定 | 明示的に許可 | 低 |
| BFF経由 | Same-Originにする | 最低 |

### 設計判断基準

- 認証が必要なAPIは `*` を使わない
- 本番環境では特定オリジンのみ許可
- 可能ならBFFパターンでCORSを回避

### Reviewer観点チェックリスト

- [ ] CORSの許可オリジンは本番環境で適切に制限されているか
- [ ] credentials: 'include' を使う場合、`*` を使っていないか
- [ ] Preflightリクエストのキャッシュは適切か

## BFFパターン

### 構成比較

```
【BFFなし】
Browser → API Server
  - CORS必要
  - トークンがブラウザに露出

【BFFあり】
Browser → Next.js (BFF) → API Server
  - Same-Origin（CORS不要）
  - トークンはサーバー側で管理可能
```

### 設計判断基準

- **BFF推奨ケース**
  - セキュリティ要件が高い
  - 複数のバックエンドAPIを集約したい
  - フロントエンド用にレスポンスを加工したい

- **BFF不要ケース**
  - シンプルな構成で十分
  - レイテンシを最小化したい
  - サーバーリソースを節約したい

### Reviewer観点チェックリスト

- [ ] BFF導入の必要性は明確か
- [ ] BFFでのエラーハンドリングは適切か
- [ ] BFFがSPOFにならない設計か
- [ ] 認証情報の受け渡しは安全か

## キャッシュ戦略

### HTTPキャッシュヘッダ

| ヘッダ | 用途 |
|--------|------|
| Cache-Control: no-store | キャッシュ禁止 |
| Cache-Control: max-age=N | N秒間キャッシュ |
| ETag | 条件付きリクエスト用 |
| Last-Modified | 更新日時ベースの検証 |

### 設計判断基準

- 認証が必要なレスポンスは `no-store`
- 静的アセットは長めの `max-age`
- 動的だが変更検知可能なら `ETag`

### Reviewer観点チェックリスト

- [ ] 機密情報を含むレスポンスにキャッシュ設定がないか
- [ ] 静的アセットのキャッシュバスティングは実装されているか
- [ ] CDN利用時のキャッシュ設計は適切か
